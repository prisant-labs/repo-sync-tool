# E-06 Implementation Plan

## Approach

Freeze the seam in the right order: types first, signatures second, codegen third, guardrails last. The core/shell split is the load-bearing constraint throughout: the payload STRUCTS and `AppError` reference live in `reposync-core::ipc` (serde + `specta::Type` only, Tauri-free), while the command and event SIGNATURES - the `#[tauri::command]` functions and the `tauri_specta::Event` types - live in `src-tauri`, because those macros require Tauri and would break the no-Tauri hygiene gate if they landed in core. So: define every payload struct in `reposync-core::ipc`, then declare the command and event signatures in `src-tauri` (`commands/`, `events.rs`) as the exact grouped signatures the brief names (with the list command normalized to `repo_list`), then wire the `tauri-specta` glue in `src-tauri/main.rs` to generate `src/lib/bindings.ts` behind `#[cfg(debug_assertions)]`, and finally pin the RC versions exactly, document the re-check and the fallback, and add the stale-`bindings.ts` CI check. The bodies stay stubbed (`todo!()` / `unimplemented!()` is fine) because this effort freezes the contract, not the behavior - every later backend effort fills a body behind a signature that no longer moves. The whole point is that once this lands, the frontend and the backend stop blocking each other.

## Steps

1. **Define payload structs in core.** Fill the E-01 (Foundation) `crates/reposync-core/src/ipc.rs` stub with `RepoSummary`, `RepoDetail`, `CheckResult`, `UpdateResult`, `ScanResult`, `ActivityRecord`, `DailySummary`, `Settings`, and supporting types (`RepoId`, `RepoFilter`, `ActivityFilter`, `UpdatePolicy`/`UpdateMode`, `WeeklySummary`, and the event payload structs). Derive `serde::Serialize`/`Deserialize` + `specta::Type` + `Debug`/`Clone` on each. Enumerate field sets against the authoritative DDL in `strategy-and-roadmap.md` Section 4.2 (the spec's "Enumerated payload fields" section lists them concretely): `ActivityRecord` mirrors `activity_records`; `Settings` mirrors the `settings` singleton (minus the internal `id` column); `RepoSummary` is the list-row split and `RepoDetail` is Summary plus the full `repos`/`repo_local_state`/`repo_remote_meta` fields and notes. `DailySummary`'s field shape is owned by E-11 (Summary engine) - this effort consumes it (`updated_count`, `releases_count`, `attention_count`, `no_change_count` + three item lists) and binds it to the transport. Add `specta` to `reposync-core/Cargo.toml` from the workspace table; confirm no `tauri` is dragged in (re-run the E-01 hygiene gate).
2. **Declare the command surface in the shell.** In `src-tauri/src/commands/` (NOT in core - `#[tauri::command]` requires Tauri), declare the `#[tauri::command] #[specta::specta]` function signatures grouped exactly as the brief groups them (Registry, Git ops, Quick actions, Activity/summaries, Settings). Normalize the list command to `repo_list` (singular, ratified 2026-06-19), not the brief's `repos_list`, since codegen makes these load-bearing TypeScript symbols. Each returns its core `ipc` type or `Result<T, AppError>`. Bodies are stubs that the downstream efforts replace. Signatures are the frozen artifact.
3. **Declare the event surface in the shell.** In `src-tauri/src/events.rs`, define the `tauri_specta::Event`-deriving types for `repo:state-changed`, `repo:check-started/completed`, `repo:update-started/completed`, `scheduler:tick`, `notification:fired`, `error:raised`, each referencing the event payload STRUCTS that live in `reposync-core::ipc` (the payload data is Tauri-free; only the `Event` derive binding payload to event name lives in the shell). Add typed emit-helper signatures so emit sites in later efforts are type-checked.
4. **Wire the `tauri-specta` builder.** In `src-tauri/src/main.rs`, construct `tauri_specta::Builder::new().commands(collect_commands![...]).events(collect_events![...])`. Call `.export(...)` to emit `src/lib/bindings.ts`, wrapped in `#[cfg(debug_assertions)]` so release builds do no file I/O. Mount the builder into the Tauri app so the generated `commands.*`/`events.*` are the real invoke/listen surface.
5. **Pin versions exactly (captured at scaffold time).** Set `tauri-specta` and `specta` (and `specta-typescript` if separate) to exact `=x.y.z` pins in the workspace dependency table, consistent with E-01's "pin all Tauri-related crates" version-pin AC. Because the brief flags `tauri-specta` v2 as a release candidate, the concrete version is confirmed and captured when the workspace is scaffolded (under E-01) rather than guessed in this doc; record the captured versions and the "re-check at each quarterly dependency review" action in this plan and the spec (AC5).
6. **Generate and commit `bindings.ts` (recorded decision).** Per the recorded decision in the spec (commit the file plus a stale-check CI gate, not generate-only), run a debug build / the export path to emit `src/lib/bindings.ts` and commit it so the frontend builds without first running the Rust exporter. Point the frontend at the typed `commands.*`/`events.*` instead of raw `invoke`/`listen`.
7. **Add the stale-bindings CI check.** Add a CI step that regenerates `bindings.ts` and fails if it differs from the committed file (a `git diff --exit-code` after regeneration). This is the drift guard that makes any unintended contract change loud.
8. **Document the fallback contingency.** In the spec (and a short note near the builder), document the contingency: if the RC proves unstable, replace generation with a single hand-maintained `bindings.ts` plus a CI test that round-trips a sample of each payload through `serde_json` against a TS-side schema. Wire it only if triggered; it is the contingency, not the plan.
9. **Verify.** `cargo check`, `cargo clippy --all -- -D warnings`, `cargo test`, `pnpm typecheck`, `pnpm lint`, the stale-bindings check, and the `cargo tree -p reposync-core | grep -i tauri` hygiene gate all green locally; push and confirm the matrix is green on both runners.

## Test strategy

- **The TypeScript build is the primary test.** Because `bindings.ts` is generated from the Rust source of truth, renaming a Rust field or changing a return type breaks `pnpm typecheck` - that is the guardrail this whole effort exists to install.
- **Stale-bindings CI check.** Regenerate-and-diff proves the committed `bindings.ts` matches the Rust contract on every push; it is the drift guard.
- **Serde round-trip on payloads.** A `reposync-core` test round-trips a representative instance of each payload (including `Result<T, AppError>`) through `serde_json` so the wire shapes are stable and lossless, independent of the codegen tool. This same test is the seed of the fallback's contingency test.
- **Dependency-hygiene gate (inherited from E-01).** Asserts `tauri-specta` did not leak into `reposync-core`; only `specta` is allowed in core.
- No behavior is tested here (bodies are stubs); the contract's shape and stability are what is under test.

## Files / modules touched

- `crates/reposync-core/src/ipc.rs` (all payload structs AND the event payload structs, Tauri-free), `crates/reposync-core/Cargo.toml` (add `specta`), `crates/reposync-core/src/lib.rs` (export `ipc`).
- `src-tauri/src/commands/**` (frozen command signatures, `#[tauri::command]`), `src-tauri/src/events.rs` (the `tauri_specta::Event`-deriving types over the core payload structs + emit-helper signatures), `src-tauri/src/main.rs` (the `tauri_specta::Builder` + `collect_commands!`/`collect_events!` + `.export` glue).
- `src-tauri/Cargo.toml` and the workspace dependency table (exact pins for `tauri-specta`, `specta`).
- `src/lib/bindings.ts` (committed generated output) and the frontend modules that import `commands.*`/`events.*`.
- `.github/workflows/ci.yml` (the stale-bindings regenerate-and-diff step).

## Risks and mitigations

- **`tauri-specta` v2 is a release candidate.** It could change API or prove unstable. Mitigation: pin `tauri-specta` and `specta` to exact versions, re-check each quarterly dependency review, and hold the documented fallback (hand-maintained `bindings.ts` + serde round-trip CI test) in reserve. Confirm the current published versions at scaffold time per the brief's caveat.
- **`bindings.ts` goes stale.** A committed generated file drifts from the Rust source. Mitigation: the regenerate-and-diff CI check fails the build on any drift (AC6).
- **Tauri leaks into core via the command/event macros.** Putting the `#[tauri::command]` signatures, the `tauri_specta::Event` types, or the builder in `reposync-core` would drag `tauri`/`tauri_specta` into the no-Tauri crate and break the hygiene gate. Mitigation: only the payload structs, event payload structs, and `AppError` stay in `reposync-core` (serde + `specta::Type` only); every Tauri-touching artifact - command signatures, event-binding types, the builder, the `.export()` - lives in `src-tauri`; the E-01 hygiene gate enforces it.
- **`RepoSummary`/`RepoDetail` field-split churn.** Getting the summary-vs-detail field boundary wrong forces later contract revisions. Mitigation: the split is enumerated concretely against the DDL (`repos` + `repo_local_state` + `repo_remote_meta`) in the spec up front - Summary is the list-row essentials, Detail is Summary plus the full state fields and notes; remaining tweaks are additive and caught by the stale-bindings check.
- **Premature freeze before payload shapes are known.** Some payloads depend on the data model (E-02) and git engine (E-03) shapes. Mitigation: the brief already names the data model (Section 4.5) and the command surface (Section 4.4), so the shapes are knowable now; the tracer bullet (E-12) exercises `repo_add_path`/`repo_check_now` against the real contract early to surface any gap while the surface is still small.

## Definition of done

All seven acceptance criteria checked: payload structs (and event payload structs) in `reposync-core::ipc` deriving `serde` + `specta::Type` with fields enumerated against the DDL; the command and event SIGNATURES frozen in `src-tauri` (`commands/`, `events.rs`) as the brief's grouped signatures with the list command normalized to `repo_list` and `Result<T, AppError>` on fallible commands; `tauri-specta` generating both commands and events into `src/lib/bindings.ts` behind `#[cfg(debug_assertions)]`; exact version pins captured at scaffold time plus the documented re-check and fallback; the committed `bindings.ts` with a stale-check CI gate; and the no-Tauri hygiene gate green. CI green on both runners and the branch ready for self-merge per the visibility-tiered policy in `EXECUTION.md`. The seam is frozen: backend and frontend can now proceed independently.
