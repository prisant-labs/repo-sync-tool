# AGENTS.md - RepoSync

Instructions for any AI agent working in this repo (Claude, Codex, or otherwise). This is the
open-standard entry point; `CLAUDE.md` at the repo root is a thin Claude-specific overlay on
top of this file and does not repeat it.

## Project map

RepoSync is a Windows-first, cross-platform desktop tray utility (Tauri v2 + React 19) that
keeps a personal library of cloned-but-not-actively-developed Git repos fresh and visible. Three
layers, one seam:

- **`crates/reposync-core`** - the headless logic crate. Git engine, scheduler, update-policy
  engine, persistence, activity log, summary engine, GitHub client, and the IPC payload types
  and error taxonomy. **Never imports `tauri`, even transitively.** That invariant is what keeps
  this crate unit-testable without a webview and is enforced in CI as a dependency-hygiene gate
  (`cargo tree -p reposync-core` must show no `tauri`).
- **`src-tauri`** - the thin Tauri shell: IPC command handlers, event emission, the system tray,
  windows, the opener (open-in folder/terminal/editor/remote), and OS-plugin wiring (autostart,
  notifications). This is the "edge." It hosts `reposync-core`; it does not contain product logic.
- **`src/`** - the React 19 + TypeScript GUI. Screens (`src/screens`), components
  (`src/components`), and the data/IPC layer (`src/lib`).

The typed IPC seam is `src/lib/bindings.ts`, generated by `tauri-specta` from the Rust command
and event types in `reposync-core::ipc`. **Never hand-edit `bindings.ts`** (it carries a
"generated, do not edit" header); regenerate it by building `src-tauri` after changing the IPC
contract. UI code only ever imports `commands` and event types from `bindings.ts`, never `invoke`
directly.

## Build and verify commands

Frontend:

```sh
pnpm typecheck   # tsc --noEmit, app + node config
pnpm lint        # eslint .
pnpm build       # vite build
```

Rust:

```sh
cargo clippy --workspace --all-targets -- -D warnings
cargo test -p reposync-core                 # prefer scoped runs; see below
cargo test -p reposync-core <filter>        # narrow further while iterating
```

Tests are tiered (since 2026-07-04; see `docs/internal/release-plans/plan_v0.9.0/ci-plan.md`
Section 3.4 decision note). The fast tier is the default: `cargo test --workspace` (or
`-p reposync-core`) skips the 27 `#[ignore]`-marked git-CLI fixture tests and completes in a
few minutes. The slow tier runs them: `cargo test -p reposync-core --features test-support
--lib -- --ignored`, plus the three feature-gated integration binaries via
`cargo test -p reposync-core --features test-support --test git_fixture_cross_check
--test policy_fixture_matrix --test scheduler_integration -- --include-ignored`.
Run the slow tier at phase gates, not inner-loop iteration; CI runs it as its own lane.

## Hard conventions

- **No em-dashes (U+2014) or en-dashes (U+2013), anywhere:** code, comments, docs, commit
  messages, JSON, YAML. Use " - " (space hyphen space) or restructure the sentence. Plain
  hyphens for numeric ranges ("2-5"). This is enforced by a pre-commit hook in some agent setups;
  treat it as a hard rule regardless.
- **Reference IDs carry a human-readable handle on first use.** Write "E-13 (tray native menu)"
  or "BL-NI-22 (O(N) group filter)", not a bare ID. Link to the doc that owns the ID where one
  exists.
- **Acceptance criteria live in specs only.** `plan_vX.Y.Z.md` release plans and
  `program-roadmap.md` aggregate and sequence; they never invent or restate AC as if authoritative.
  The spec's own AC is the source of truth.
- **`docs/internal/` is tracked; `_LOCAL/` is gitignored.** Durable project docs go under
  `docs/internal/`. Scratch work, audits, session logs, and prototypes go under `_LOCAL/` and are
  never committed.
- **Commit style:** short imperative subject with a scope prefix matching recent history (for
  example `feat(scheduler): ...`, `docs(backlog): ...`, `fix(opener): ...`); see `git log` for the
  live convention. Co-authored-by trailer on agent commits per `EXECUTION.md`.
- **Never commit unless asked.** Leave changes in the working tree; the human or an explicit
  instruction triggers `git commit`.

## The shell-crate chokepoint

Anything that compiles `src-tauri` (Rust edge code, IPC command signatures, event definitions,
plugin wiring) is a strict serial chain: only one such change builds at a time, because
`cargo build`/`clippy` against `src-tauri` is the slow, contended step and concurrent edits there
race. Work confined to `reposync-core` only, or to `src/` (frontend) only, parallelizes freely
against everything else, including against each other. When planning concurrent agent work,
default to serializing anything that touches `src-tauri/**`, and freely parallelize
`crates/reposync-core/**`-only and `src/**`-only changes.

## Doc-update-in-change cadence

Docs move with the code, in the same change, not as a follow-up. See `docs/README.md` for the
full cadence; the short version:

1. Any change to architecture, the IPC contract, scope, or user-facing behavior updates the
   relevant living doc (`docs/faq.md`, `docs/architecture.md`, `docs/explanation.md`) and the
   effort's own `spec.md` Task Summary block in the same change.
2. Anything not immediately actioned (new ideas, deferred work, review findings) goes to
   `docs/backlog.md`, not left undocumented.
3. Feature-tracking docs (`docs/internal/release-plans/plan_v0.9.0/feature-inventory.md`,
   `edge-wiring-plan.md`) stay current with what actually shipped, not what was planned to ship.
   A doc that lags the code is a bug, not a follow-up.
